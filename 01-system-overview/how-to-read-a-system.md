# How To Read A System

## 它解决什么

读论文、系统或 demo 时，最重要的是先知道该看什么。这个页面给一个固定读法：先定位系统解决哪个技术环节，再判断证据强度。

## 一页读懂法

看到一个系统、论文或项目时，先填这张卡：

| 问题 | 要记录什么 |
| --- | --- |
| 它要完成什么任务？ | 抓取、导航、长程家务、移动操作、仿真评测等 |
| 输入是什么？ | 图像、语言、深度、点云、触觉、本体状态 |
| 输出是什么？ | 动作 token、轨迹、末端位姿、关节控制、技能调用 |
| 数据从哪来？ | 遥操作、仿真、人类视频、机器人自采集、混合数据 |
| 模型学什么？ | 感知表征、策略、世界模型、规划器、控制器 |
| 在哪里评测？ | 仿真、实验室实机、公开 benchmark、真实部署 |
| 证据强不强？ | 成功率、任务数量、对照实验、代码、数据、失败分析 |
| 局限是什么？ | 场景窄、机器人窄、任务短、需要人工干预、成本高 |

## 技术环节定位

- 感知改进：更好识别物体、场景、空间关系。
- 表征改进：更好表示状态、动作、对象、世界变化。
- 数据改进：更大规模、更高质量、更可复用。
- 策略改进：更好从观察和目标生成动作。
- 规划控制改进：更好分解任务、避障、执行、恢复。
- 仿真评测改进：更好训练、比较、复现。
- 部署改进：更安全、更实时、更便宜、更稳定。

## 证据强度阶梯

1. 观点或概念图：适合启发，不足以下结论。
2. 选择性 demo：说明可能性，不说明稳定性。
3. 受控实验：可以比较方法，但要看设置是否窄。
4. 公开 benchmark：更可比较，但可能与真实任务有距离。
5. 多场景实机评测：更接近真实能力。
6. 长期部署数据：最有价值，也最少见。

## 常见误读

- 把“能完成一个任务”看成“具备通用能力”。
- 把“能听懂语言”看成“能稳定执行动作”。
- 只看模型结构，不看数据来源和动作接口。
- 只看成功视频，不看试验次数和失败类型。

## 建议输出

读完一篇材料后，不要只写摘要。至少写下：

- 我学到了哪个技术环节的新东西。
- 它解决了什么瓶颈。
- 它的证据有多强。
- 它还不能说明什么。
