# 机器人运动学基础：正解、逆解和坐标系

## 它解决什么问题

机器人运动学（Robot Kinematics）解决的是：机器人身体怎么动，末端执行器会到哪里。

它先不关心电机力矩、摩擦、重量和加速度，只关心几何关系：

- 每个关节转了多少？
- 每段机械结构有多长？
- 末端执行器现在在哪里、朝向哪里？
- 如果想让夹爪到某个位置，关节应该怎么转？

对 0 基础读者，可以先把机械臂想成“很多段尺子用转轴接起来”。每个转轴转一点，最末端的位置就会变化。

## 最小例子：机器人去拿杯子

任务：让机械臂抓桌上的杯子。

系统至少要知道三件事：

1. 杯子在哪里：来自相机、深度、点云或 6D 位姿估计。
2. 夹爪现在在哪里：来自机器人关节状态和正向运动学。
3. 夹爪怎么到杯子那里：需要逆向运动学、运动规划和控制。

如果只知道杯子在图像里的像素位置，还不够。机器人最终要控制的是关节角度、电机速度或末端轨迹，所以需要把“空间目标”转成“身体动作”。

## 先认识几个词

| 术语 | 英文 | 直观解释 |
| --- | --- | --- |
| 连杆 | Link | 机械臂的一段刚性结构，像手臂里的上臂、前臂。 |
| 关节 | Joint | 两段连杆之间可以运动的连接点，例如旋转关节或直线滑轨。 |
| 自由度 | Degree of Freedom, DoF | 机器人可以独立运动的维度。一个旋转关节通常算 1 个自由度。 |
| 末端执行器 | End Effector | 真正执行任务的末端工具，例如夹爪、吸盘、焊枪、手指。 |
| 位姿 | Pose | 位置 + 朝向。位置回答“在哪里”，朝向回答“面向哪里”。 |
| 工作空间 | Workspace | 机器人末端执行器能够到达的空间范围。 |
| 关节限制 | Joint Limit | 每个关节能转动或移动的最大范围。 |
| 关节空间 | Joint Space | 用每个关节的位置描述机器人状态。 |
| 笛卡尔空间 | Cartesian Space | 用真实空间里的 x、y、z 位置和朝向描述目标。 |
| 机器人模型 | Robot Model | 描述机器人连杆、关节、坐标系和限制的结构文件或参数集合。 |

一个 6 自由度机械臂通常可以在 3D 空间里控制末端的位置和朝向；如果自由度更多，例如 7 自由度，就会有更多姿态选择，也更容易绕开障碍物。

## 两种空间：关节空间和笛卡尔空间

机器人最容易混淆的一点是：目标往往在真实空间里描述，但机器人身体通常由关节控制。

关节空间（Joint Space）像是在说：

```text
第 1 个关节：30 度
第 2 个关节：-20 度
第 3 个关节：75 度
...
```

笛卡尔空间（Cartesian Space）像是在说：

```text
夹爪位置：x = 0.45 米, y = -0.10 米, z = 0.30 米
夹爪朝向：向下，略微朝杯子倾斜
```

正解把关节空间转成笛卡尔空间；逆解把笛卡尔空间目标转回关节空间。

## 常见关节类型

入门阶段先记住两类就够：

| 关节类型 | 英文 | 它怎么动 | 例子 |
| --- | --- | --- | --- |
| 旋转关节 | Revolute Joint | 绕一根轴转动 | 机械臂肩、肘、腕关节 |
| 移动关节 | Prismatic Joint | 沿一条直线滑动 | 升降台、直线滑轨 |

多数工业机械臂主要由旋转关节组成。移动机器人、人形机器人、灵巧手也可以用同样的运动学思想分析，只是身体结构更复杂。

## 坐标系：机器人世界里的“尺子和方向”

机器人系统里通常有多个坐标系：

- 世界坐标系：整个场景的参考系。
- 机器人基座坐标系：以机器人底座为原点。
- 关节坐标系：每个关节自己的局部参考系。
- 末端执行器坐标系：夹爪或工具自己的参考系。
- 相机坐标系：以相机为原点看到的 3D 空间。

同一个杯子，在相机坐标系里可能是 `(0.2, 0.1, 0.8)`，在机器人基座坐标系里可能是 `(0.5, -0.3, 0.2)`。数值不同，不代表杯子变了，只是“用哪把尺子量”不同。

很多真实机器人抓偏，不是因为模型没识别出物体，而是因为坐标系转换错了、相机标定不准，或者目标位姿没有转到机器人能理解的坐标系。

## 变换：把一个坐标系里的点换到另一个坐标系

坐标变换（Transform）解决的是：

> 同一个点，换一套坐标系应该怎么表示？

例如相机看到杯子在相机前方 0.8 米，但机器人要知道杯子相对自己底座在哪里。系统需要一个“相机坐标系 -> 机器人基座坐标系”的变换。

工程里经常用变换矩阵（Transform Matrix）或位姿表示来描述这件事。0 基础阶段不用急着推矩阵公式，先记住：

- 平移：原点换了，数值会整体偏移。
- 旋转：方向轴换了，x/y/z 的意义会变。
- 组合：真实机器人通常要连续经过多个坐标系。

一个典型链路是：

```text
杯子在相机坐标系的位置
-> 转到机器人基座坐标系
-> 变成夹爪应该到达的目标位姿
-> 用 IK 求关节角度
```

## 朝向怎么表示

位置比较好理解：x、y、z 三个数字就能描述一个点在哪里。朝向更麻烦，因为机器人末端可能需要“从上往下抓”“侧着插入”“沿杯柄方向靠近”。

常见朝向表示有三类：

| 表示 | 直观理解 | 入门阶段怎么记 |
| --- | --- | --- |
| 欧拉角 | 用绕几个轴旋转的角度描述朝向。 | 好理解，但有些姿态会不稳定。 |
| 旋转矩阵 | 用 3 个方向轴描述一个坐标系的朝向。 | 工程和几何计算常见。 |
| 四元数 | 用 4 个数紧凑表示 3D 朝向。 | 机器人和图形系统常用，适合插值。 |

刚入门不用急着掌握它们的公式。先记住：位姿不是只有位置，朝向也是任务成功的一部分。夹爪到达杯子旁边但朝向不对，仍然可能抓不起来。

## 机器人模型从哪里来

正解和逆解都需要知道机器人身体结构。系统通常不会凭空知道机械臂每段多长、每个关节怎么转，而是从机器人模型中读取。

常见描述方式：

- URDF：机器人领域常见的结构描述文件，记录连杆、关节、坐标系、碰撞几何等信息。
- DH 参数：一种经典机械臂建模方式，用一组参数描述相邻连杆和关节的几何关系。
- 厂商模型：机器人厂商提供的模型、驱动和标定参数。

如果机器人模型不准，正解算出来的末端位姿会偏；如果关节限制或碰撞几何不准，IK 和运动规划也会做出不可靠的结果。

## 正向运动学：已知关节，求末端

正向运动学（Forward Kinematics, FK），也常被叫作正解。

它回答的问题是：

> 如果每个关节现在是这些角度，夹爪在哪里、朝向哪里？

输入：

- 每个关节的位置，例如 6 个关节角度。
- 机器人结构参数，例如每段连杆长度、关节安装方向。

输出：

- 末端执行器的位姿，也就是位置和朝向。

一个简单二维机械臂可以这样理解：

```text
肩关节转 30 度
肘关节转 60 度
两段手臂长度已知
=> 手掌的位置可以用几何算出来
```

正解通常比较确定：同一台机器人、同一组关节角度，末端位置一般只有一个。

## 和 HTML 演示对照：二维 FK 公式

演示中的二维机械臂只有两个旋转关节，所以可以直接用平面三角关系计算末端位置：

```text
x = L1 cos(theta1) + L2 cos(theta1 + theta2)
y = L1 sin(theta1) + L2 sin(theta1 + theta2)
```

这里要注意两点：

- `theta1` 是第一段连杆相对世界坐标系的角度。
- `theta1 + theta2` 才是第二段连杆在世界坐标系里的角度。

如果你在演示里把肩关节设为 `35°`，肘关节设为 `65°`，第二段连杆不是只按 `65°` 指向，而是按 `35° + 65° = 100°` 指向。这就是很多初学者第一次看 FK 时容易混淆的地方：后一个关节的角度通常是相对上一段连杆定义的。

## 正解为什么通常比较容易

正解是沿着机器人身体结构“一段一段往外算”：

```text
基座
-> 第 1 个关节和连杆
-> 第 2 个关节和连杆
-> 第 3 个关节和连杆
-> ...
-> 末端执行器
```

每一段只需要知道“上一段在哪里”和“这一段怎么转、怎么平移”。所以给定机器人模型和关节状态，末端位姿通常能稳定算出来。

## 逆向运动学：已知末端，求关节

逆向运动学（Inverse Kinematics, IK），也常被叫作逆解。

它回答的问题是：

> 如果想让夹爪到这个位置和朝向，每个关节应该转多少？

输入：

- 目标末端位姿，例如“夹爪到杯子右侧 10 厘米，并朝向杯口”。
- 机器人当前关节状态。
- 关节限制、碰撞限制、姿态偏好等约束。

输出：

- 一组或多组可行的关节角度。

逆解比正解更难，因为它可能出现几种情况：

| 情况 | 含义 | 例子 |
| --- | --- | --- |
| 无解 | 目标根本够不到，或姿态不可能实现。 | 杯子太远，超出机械臂工作空间。 |
| 一个解 | 只有一种关节姿态能到达。 | 约束很强，路径很窄。 |
| 多个解 | 多种关节姿态都能让夹爪到同一个地方。 | 手肘可以朝上，也可以朝下。 |
| 近似解 | 到不了完全目标，只能尽量靠近。 | 目标朝向太苛刻，但位置可以到。 |

这就是为什么 IK 不只是“代公式”。真实系统常常要在多个候选解里选一个更安全、更平滑、更不容易碰撞的。

## 和 HTML 演示对照：二维 IK 公式

演示里的 IK 是一个解析解例子。目标点是 `(x, y)`，到基座的距离是 `d`：

```text
d^2 = x^2 + y^2
cos(theta2) = (d^2 - L1^2 - L2^2) / (2 L1 L2)
theta1 = atan2(y, x) - atan2(L2 sin(theta2), L1 + L2 cos(theta2))
```

这组公式背后的直觉是：先看基座、肘关节、目标点形成的三角形，再用三角关系求肘关节和肩关节。

演示还刻意展示了三个真实系统里很常见的问题：

| 现象 | 在演示里怎么看 | 对真实机器人的启发 |
| --- | --- | --- |
| 可达范围有限 | 目标点会被限制在两个虚线圆之间 | 机器人有工作空间，不是屏幕上任意点都能到 |
| IK 可能多解 | 灰色机械臂显示另一种姿态 | 同一个末端位置可能对应“肘上 / 肘下”等不同关节姿态 |
| 目标要满足几何约束 | `cos(theta2)` 必须落在 `[-1, 1]` 附近 | 超出几何约束时，解析 IK 会无解，数值 IK 也只能近似 |

由于 `L1 = 1.00`、`L2 = 0.75`，这个二维机械臂的目标距离必须大致满足：

```text
|L1 - L2| <= d <= L1 + L2
0.25 <= d <= 1.75
```

外圈 `1.75` 表示手臂完全伸直时最远能到哪里；内圈 `0.25` 表示两段连杆长度不同，末端也不能无限靠近基座。真实机械臂还会额外受到关节角度、碰撞、线缆、速度和力矩限制，所以可达空间通常比这个理想几何范围更小。

## 逆解的两种常见求法

逆解大致有两类思路：

| 方法 | 直观解释 | 优点 | 局限 |
| --- | --- | --- | --- |
| 解析解 | 通过几何或公式直接算出关节角。 | 快、稳定、结果清楚。 | 通常只适合结构比较规则的机器人。 |
| 数值解 | 从一个初始姿态开始，反复调整关节，让末端越来越接近目标。 | 灵活，能处理复杂约束。 | 可能慢、可能陷入局部最优，也依赖初始姿态。 |

很多实际系统会优先用成熟 IK 求解器，而不是手写公式。读论文或项目时，要看清楚它是自己预测关节动作，还是把末端目标交给 IK 求解器。

## 冗余：为什么 7 自由度机械臂更灵活

如果机器人自由度刚好等于任务需要的维度，目标位姿可能只有少数解。  
如果机器人自由度更多，就叫冗余（Redundancy）。

冗余不是坏事。它意味着机器人可以在“末端不动”的情况下，调整肘部或身体姿态：

- 避开障碍物。
- 远离关节极限。
- 选择更自然、更省力的姿态。
- 给后续动作留下空间。

人类手臂也有类似冗余。你可以让手掌停在同一个位置，但肘部朝上、朝下或向外，姿态都不一样。

## 正解和逆解的关系

可以把它们看成两个方向相反的问题：

```text
正解 FK：关节角度 -> 末端位姿
逆解 IK：末端目标 -> 关节角度
```

抓杯子的流程里，两者都会出现：

1. 读取当前关节角度。
2. 用正解算出夹爪当前位姿。
3. 感知系统给出杯子位姿。
4. 设定一个抓取目标位姿。
5. 用逆解求一组可行关节角度。
6. 用运动规划检查从当前姿态到目标姿态是否碰撞。
7. 控制器执行轨迹。
8. 执行中继续用正解和传感器反馈修正。

## 为什么逆解经常失败

新手很容易以为“目标点在前面，机械臂就应该能到”。真实情况更挑剔。

常见失败原因：

- 目标超出工作空间。
- 目标位置能到，但目标朝向不能满足。
- 某个关节超过角度限制。
- 手臂到目标的姿态会碰到桌子、墙或自己。
- 目标附近有奇异点，关节速度会突然变得很大。
- 初始姿态不好，数值求解器陷入局部最优。
- 相机坐标系到机器人坐标系的标定有误。

所以 IK 通常要和碰撞检测、运动规划、约束优化、控制器一起使用。

## 雅可比和奇异点：先知道直觉就够

雅可比矩阵（Jacobian）描述的是：

> 关节小小动一下，末端会怎么动？

它把“关节速度”联系到“末端速度”。例如肩关节转一点，夹爪可能移动很多；手腕关节转一点，夹爪位置可能几乎不变，但朝向变了。

奇异点（Singularity）可以先理解成“机器人姿态变得很别扭的位置”。在奇异点附近，想让末端朝某个方向平稳移动，某些关节可能需要非常大的速度，或者某些方向根本动不了。

直观例子：

- 手臂完全伸直时，再想继续往前伸就很困难。
- 某些关节轴线重合时，机器人会失去一部分可控方向。

工程上会尽量避开奇异点，或者在 IK 和轨迹规划里加入惩罚项。

## 位置、速度和轨迹

控制机器人时，动作不一定只是一组目标角度。常见层次包括：

| 层次 | 问题 | 例子 |
| --- | --- | --- |
| 位置 | 最终到哪里？ | 夹爪到杯子旁边。 |
| 速度 | 现在往哪个方向、多快移动？ | 夹爪以 5 cm/s 靠近杯子。 |
| 轨迹 | 从现在到目标的整段路径怎么走？ | 2 秒内绕开障碍物并到达抓取位姿。 |

正解和逆解常常处理位置关系；雅可比更多出现在速度层面；运动规划则更关心整段轨迹是否安全、平滑、可执行。

## 运动学、运动规划和控制的区别

这三个词经常混在一起，但它们解决的问题不同：

| 模块 | 主要问题 | 例子 |
| --- | --- | --- |
| 运动学 | 身体几何关系是什么？ | 关节角度对应哪个末端位姿？ |
| 运动规划 | 怎么安全地从 A 到 B？ | 机械臂如何绕开桌面和障碍物？ |
| 控制 | 怎么让真实机器人跟上轨迹？ | 电机如何稳定执行目标关节角？ |

再往下还有动力学（Dynamics），它会关心力、质量、惯性、加速度、接触和摩擦。对入门读者，先把运动学和运动规划分清楚，再看动力学会更顺。

## 在具身智能系统里的位置

VLA、Diffusion Policy、Flow Policy 等学习型策略常常输出动作，但这些动作最终仍要落到机器人的身体上。

几种常见接口：

- 输出关节位置：模型直接预测下一组关节角。
- 输出末端位姿：模型预测夹爪目标，再用 IK 转成关节角。
- 输出末端速度：模型预测夹爪移动方向，再用雅可比或控制器执行。
- 输出轨迹：模型预测一段未来动作，再由控制器逐步跟踪。
- 输出技能参数：模型选择“抓取杯子”，底层技能内部处理 IK、规划和控制。

这也是为什么懂一点 FK/IK 很重要：它能帮助你看懂论文或系统到底是在控制“关节”、控制“末端”，还是只是在给底层控制器发目标。

## 常见误区

- 误区 1：能检测到物体，就一定能抓到。  
  感知只是给目标，机器人还要能到达、避障并稳定执行。

- 误区 2：IK 有解，就一定能安全执行。  
  IK 只说明某个姿态可能可达，不保证路径中不碰撞。

- 误区 3：末端位置对了，任务就成功。  
  抓取还需要朝向、夹爪开合、接触力、摩擦和时序。

- 误区 4：仿真里能动，真实机器人就能动。  
  真实系统有标定误差、延迟、柔性、摩擦和控制误差。

- 误区 5：演示里的目标自动贴到可达范围，所以真实系统也会自动修正。  
  HTML 演示为了教学会把目标限制在可达圆环内；真实系统需要显式做目标裁剪、失败报告、重新规划或人工接管。

## 读论文时怎么用这页

看到机器人论文时，可以问：

- 动作空间是关节空间，还是末端空间？
- 如果模型输出末端位姿，谁负责 IK？
- 有没有碰撞检测和运动规划？
- 是否考虑关节限制、速度限制和奇异点？
- 评测是在仿真、真实机器人，还是只展示 demo？
- 失败时是感知错了、IK 无解、规划失败，还是控制没跟上？

这些问题能帮你把“模型效果好不好”拆成更具体的工程原因。

## 基础概念覆盖清单

| 概念 | 你应该记住什么 |
| --- | --- |
| 连杆和关节 | 机器人身体由刚性连杆和可运动关节组成。 |
| 自由度 | 机器人能独立控制的运动维度。 |
| 位姿 | 位置 + 朝向，不只是一个点。 |
| 坐标系 | 同一个物体在不同坐标系下数值不同。 |
| 坐标变换 | 把相机看到的位置转成机器人能执行的目标。 |
| 关节空间 | 用关节角或关节位移描述机器人状态。 |
| 笛卡尔空间 | 用真实空间的位置和朝向描述目标。 |
| 朝向表示 | 欧拉角、旋转矩阵和四元数都是描述朝向的方式。 |
| 机器人模型 | FK/IK 需要知道连杆长度、关节方向、坐标系和限制。 |
| 正解 FK | 已知关节，求末端在哪里。 |
| 逆解 IK | 已知末端目标，求关节怎么摆。 |
| 工作空间 | 机器人末端能到达的范围。 |
| 可达圆环 | 二维两连杆里由最短距离和最长距离形成的可达区域，HTML 演示用虚线圆展示。 |
| 关节限制 | 每个关节都有可动范围，不能无限转。 |
| 多解和无解 | IK 可能有多个姿态，也可能根本够不到。 |
| 冗余 | 多出来的自由度能帮助避障和选择更好姿态。 |
| 雅可比 | 描述关节小变化如何影响末端运动。 |
| 奇异点 | 某些姿态下机器人会失去可控方向或需要极大关节速度。 |
| 运动规划 | 在 IK 之外，还要检查路径是否碰撞、是否平滑。 |
| 控制 | 让真实电机稳定跟踪目标，而不是只停留在几何计算。 |

## 代表工具和项目

- MoveIt 2：机械臂运动规划、IK、碰撞检测和执行框架。
- cuRobo：GPU 加速的 IK、轨迹优化和运动生成工具。
- Pinocchio / Drake / KDL：常用于机器人运动学、动力学和控制建模的工具。
- URDF：描述机器人结构、连杆、关节和坐标系的常见格式。
