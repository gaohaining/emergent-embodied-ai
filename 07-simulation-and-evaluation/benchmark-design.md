# Benchmark Design

## 它解决什么

Benchmark 用来比较不同方法，但设计不好会误导判断。读 benchmark 时需要知道它测的到底是什么，没测什么。

## 一个好 benchmark 应该说明

- 任务是什么。
- 环境是仿真还是真实。
- 输入和输出是什么。
- 使用什么机器人平台。
- 训练集和测试集如何划分。
- 是否包含新物体、新场景、新指令。
- 指标是什么。
- 是否允许人工干预。
- 失败如何记录。

## 常见指标

| 指标 | 含义 | 局限 |
| --- | --- | --- |
| 成功率 | 任务完成比例 | 不说明效率和安全 |
| 平均耗时 | 完成任务所需时间 | 快不一定稳 |
| 动作步数 | 动作效率 | 不同动作接口难比较 |
| 碰撞次数 | 安全风险 | 需要清晰定义 |
| 人工接管次数 | 自主性 | 接管标准可能不同 |
| 泛化成功率 | 新场景表现 | 测试分布设计很关键 |

## 评测层次

1. 单技能评测。
2. 多任务评测。
3. 新物体或新场景泛化。
4. 长程任务。
5. 干扰和失败恢复。
6. 长期部署。

## 读 benchmark 时先问

- 任务是否接近真实需求？
- 测试集是否和训练集足够不同？
- 是否公开失败案例？
- 是否有真实机器人结果？
- 是否有消融实验说明改进来自哪里？

## Open Questions

- 通用机器人策略是否需要统一评测协议？
- 长期部署能力如何标准化评测？
- 如何设计不容易被数据记忆污染的开放任务？
